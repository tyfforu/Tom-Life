## 一、上下文管理

### 背景

很多人把 token 用在刀背上：
- 在 context 快满的情况下还在疯狂输出
- 装了 skill 但不知道有没有用上
- 上下文污染导致模型表现变差

📌 **关键提示**：当上下文窗口占到 **30%-40%** 时，就会出现一定程度的上下文腐烂（之前作者自己用到 60% 才开始管理）。

### 常用三个命令

#### 1. `/compact` - 压缩对话

**适用场景**：
- 任务跑了很长时间，进行过多轮对话
- 上下文窗口占到 50% 以上
- 发现大模型开始回答奇奇怪怪的答案

**作用**：把前面的对话压缩成一个摘要，清掉冗余信息只保留关键的，让 context 变得干净，后续表现回归正常。

**作者习惯**：
- 同一个任务超过 20-30 轮对话 → 自压缩
- 上下文窗口 50% 以上 → 自压缩
- 模型回答异常 → 先 compact

#### 2. `/clear` - 清空对话

**适用场景**：
- 一个任务已经完全跑偏
- 想换一个新任务

**对比**：
- `compact` = 整理桌面
- `clear` = 清空桌面重新来

选择取决于当前任务是否有值得保留的上下文。

#### 3. `/by-the-way` - 侧线讨论

**适用场景**：不想侵入当前上下文窗口。

**例子**：
- 正在 web coding，但想到产品逻辑有问题 → 用 `by the way` 讨论，不记录在上下文
- 用 `by the way` 把需求记录在某个文档里，等当前任务结束后，再调用文档里的需求继续工作

### 个人使用习惯

1. **引用文件时指定路径和文件名**
   - 不要让大模型自己扫描文件/扫描整个仓库
   - 否则它可能写脚本去找文件，token 消耗得不偿失

2. **长任务或复杂任务用 Plan Mode**
   - 可以大幅度减少 token 消耗

3. **尽量让 Cloud Code 完成一整个工作流**
   - 而不是一步一步告诉它做什么
   - Cloud Code 是强大的 agent，能从第一步到最后一步完美执行
   - 如果每一步都拆开 → 上下文变长 → 幻觉、丢失、污染

## 二、Prompt 缓存机制

### 缓存原理

Cloud 的 prompt 缓存机制：
- 如果上一次请求的内容与这一次请求的前缀相同
- Cloud 就不会重新处理那段内容，直接用缓存
- **速度更快，token 消耗大幅降低**

在 Cloud Code 里：
- `Cloud.md` 的内容
- 项目文件的内容
- 在同一会话里可以被缓存

### 缓存失效场景

#### 1. 修改 `Cloud.md` 文件

- `Cloud.md` 在缓存架构中被视为一个整体模块
- 位于缓存前缀的中间位置
- **一旦改了文件中哪怕一个标点符号**
- 系统会判定从 `Cloud.md` 这个文件的模块开始，到后续所有内容（包括历史对话的缓存）都会全部失效

#### 2. 对话间隔太长

- Cloud 的缓存默认有 **5 分钟的超时时间**
- 超过 5 分钟没有新的请求 → 缓存失效
- **保持对话的节奏比长时间等待更好**

📌 **建议**：每次离开之前，先 compact 一下是最好的。

### 实践建议

理解这个机制后，应该有意识地去组织 `Cloud.md` 文件的结构，让 Cloud Code 能够持续用到缓存 → 整体速度明显更快，token 使用量更少。

## 三、常用 Skills

> Skill 是 Cloud Code 的可安装能力包，把一套提示词和逻辑打包成一个命令，以后就可以直接调用。

### 1. `planning-with-file`

**适用场景**：复杂任务，不想让 Cloud Code 直接开始乱动。

**工作流程**：
- 把任务拆解成结构化的计划
- 写进一个文件里
- 你 review 确认之后
- 它再按照文件里的计划一步步执行

**作者使用频率最高**，甚至所有偏复杂的任务都会先用这个 skill。

**实际应用例子**（Web Coding）：
- 原本流程：花一天想架构 → 花几天开发
- 现在流程：花几天跟 Cloud 聊设计 → 把想法记录到文件 → 再让 Cloud 生成代码
- 结果：代码结构、约束、规范都非常工整

### 2. 前端设计相关 Skills

- `front-end-design`
- `UI-UX-Pro`
- `remotion`（做视频基本都会用到）

**特点**：专门为前端界面设计调优过的 skill。

**建议**：
- 很多开发人员对 UI 设计不熟悉
- 用这些 skill 可以帮到很多
- 官网提供很多素材和模板
- 配合像 `stitch` 这种 AI 设计软件 → 事半功倍

### 3. `notebook-lm`

**背景**：
- 原本 Gemini 可以直接连到 Notebook LM（都是谷歌全家桶）
- 但后来 Gemini 降级比较厉害，结果不能满足要求
- 尝试把 Notebook LM 输出的结果给 Cloud Code 执行 → 效果很好
- 现在有直接可用的 skill

### 4. `everything-cloud-code`

**功能**：汇总了 Cloud Code 目前所有功能的用法，相当于随时可查的内置使用手册。

**优点**：不确定某个功能怎么用时，直接调用它，比翻官方文档快很多。

**注意**：
- 这个 skill 比较消耗 token
- 一定要去关闭它的一些 MCP
- 即使关了，token 消耗也比较大
- 没有碰到复杂任务时是用不上的
- 但整体质量非常好

### 5. `superpowers`

**特点**：
- 包含一整个完整的软件工程生命周期的 skill 集合
- 非常适合小白
- 最重要的是它的"脑爆"技能

**建议**：如果你是小白刚刚入手 web coding，用这一个 skill 就足够了。

### 6. 卡帕西的 Skill

**特点**（相较于 superpowers 和 everything-cloud-code）：
- 更像是一个靠谱的资深工程师
- 强调先清楚再动手，不乱猜
- 能简单的就不要搞得太复杂
- 改动都是以最小的成本去改
- 每一步都尽可能去做验证

**适用场景**：
- 修 bug
- 改老项目
- 做重构

## 四、Hooks 使用技巧

> Hooks 的定义：钩子，允许你自定义一些触发器，在 Cloud Code 做完某件事情的时候，会自动执行一段你所定义的脚本或命令。

### 1. 自动代码格式化

**场景**：每次让 Cloud Code 修改了代码之后，自动让它去跑 `prettier` 和 `eslint`。

**好处**：
- 不用每次提醒
- 不用担心它改完之后代码格式会乱掉

### 2. 任务结束自动通知

**场景**：跑一个时间比较长的任务时，可以做别的事情，任务完成时触发系统通知到手机。

**实际应用**：
- 之前用 OpenCloud 时，它需要不断轮询去监控 Cloud Code → 非常消耗 token
- 后来自定义了一套方案：让 OpenCloud 触发 Cloud Code 后，通过 Cloud Code 的 hook 回调来通知 OpenCloud
- 结果：减少不少 token 消耗

### 3. 自动记录日志

**场景**：工具调用前后自动记录日志。

**好处**：
- 想知道 Cloud Code 在一个任务里到底做了什么
- Hooks 可以在每次工具调用前后自动记录日志
- 任务结束后就能看到完整的执行过程

## 五、实用插件推荐

> 插件：Skills、Hooks、MCP 可以揉在一块成为一个 CI/CD 插件。

### 1. CloudHub

**功能**：
- 实时监控你自己套餐的使用量
- 查看目前这个 session 的上下文情况

**配合**：前面上下文管理的一些方式组合使用 → 效果非常好。

### 2. Figma MCP

**适用场景**：
- 配合前面讲的 `front-end-design`、`UI-UX-Pro` skills
- 特别是工作中产品用 Figma 的团队

**好处**：直接通过 MCP 方式把设计稿搞进来 → 从设计到实现顺很多。

### 3. Sentry MCP

**适用场景**：排查线上 bug。

**好处**：
- 爆出错误后，Cloud Code 能够更快结合异常信息、堆栈和上下文去定位问题
- 省掉自己去翻半天的日志
