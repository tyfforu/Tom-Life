## 1. 安装准备工作

### 系统要求

- 支持 Windows 和 Mac 两大操作系统
- **唯一欠缺功能**（Windows）：Computer Use（自动操作电脑的能力）

### 需要安装的软件

1. **Gateway** - 网关
2. **Node.js** - JavaScript 运行时
3. **VS Code** - 代码编辑器

> 💡 详细操作步骤可以参考作者上期视频《从零开始用国内网络跑通一切 AI Agent》

### 安装步骤

1. 前往 Codex 官网下载对应系统的安装包
2. 一路点击"下一步"完成安装
3. 启动 Codex APP
4. 选择 ChatGPT 账户完成登录
   - **免费账户也能使用 Codex**，不过额度比较低

### 初始设置

1. **选择工作类型**：Codex 会根据你的选择预装一些内置的插件和 Skills
2. **选择使用场景**：编程 / 日常工作（后续都可以在设置里修改）
3. **点击"设置沙盒"按钮**：完成沙盒的初始化

## 2. Codex APP 界面介绍

### 三栏布局

Codex APP 采用经典的三栏布局：

| 区域 | 功能 |
|------|------|
| **左侧** | 任务列表 |
| **中间** | 对话窗口 |
| **右侧** | 多功能区域 |

### 项目工作

**创建项目的方式**：

1. 点击"进入项目工作"
2. 选择"使用现有文件夹"
3. 选择你的项目文件夹

**项目特点**：
- 项目名称就是文件夹的名称
- 项目里面展示正在运行的任务

**快捷键**：
- Windows：`Ctrl + N` - 开启新对话
- Mac：`Command + N` - 开启新对话

## 3. 任务管理和项目工作

### 并行任务执行

Codex 支持多个任务并行执行：

- 可以在两个项目并行工作
- 还可以开启更多的工作对话
- 每个对话可以选择属于哪个项目

### 任务状态标识

| 标识 | 状态 |
|------|------|
| 转圈的小图标 | AA 正在工作 |
| 绿色标签 | 等待批准（需要权限审批） |
| 蓝色小点 | 已执行完毕 |

### 任务列表功能

1. **新对话按钮**：开启新的对话
2. **项目选择**：可以选择对应的项目，也可以选择"不使用任何项目"（纯粹的闲聊）
3. **搜索功能**（`Ctrl + G` / `Command + G`）：搜索近期的对话历史
   - ⚠️ 注意：只能搜索到对话的标题，无法搜索到对话里面的内容
   - 对话标题是 AA 根据对话内容自动摘要生成的
4. **重命名对话**：双击某个对话可以进行重命名
5. **归档对话**：点击归档按钮确认后，对话在左侧边栏消失
   - 在"设置 → 已归档对话"里面可以找到删除的对话
   - 点击"取消归档"就可以把它还原回来

## 4. 权限控制 - 沙箱机制

### Codex vs Cloud Code 的权限控制差异

| 特性 | Codex | Cloud Code |
|------|-------|------------|
| **沙箱开启** | 必须（地基） | 手动开启（额外的保护） |
| **默认权限** | 可读写沙箱内所有文件 | 需要配置 |
| **沙箱外文件** | 禁止修改 | 取决于配置 |
| **联网** | 默认禁止 | 取决于配置 |

> 🔑 **核心概念**：Codex 的沙箱是整个权限系统运行的地基，而 Cloud Code 的沙箱更像是一层可以额外开启的保护。

### 沙箱的两点默认限制

1. **不能修改沙箱外的文件**
2. **沙箱禁止联网**

> ⚠️ 这两点硬性限制并不是靠模型自觉遵守，而是 Codex 使用操作系统底层功能实现的。
> - MacOS 使用系统内置的 Seatbelt Sandbox 机制
> - 这是 **Harness Engineering 概念**的一个典型工程实现

**Harness 的比喻**：
- AA 就像一匹能力很强的马
- 沙箱、权限、审批机制这些就是套在它身上的**缰绳**（Harness）
- 用操作系统级别的机制把 AA 的能力约束在一个可控的范围之内

### 提权操作（Escalate）

如果 Codex 需要修改沙箱外文件或者需要联网，可以向用户申请权限，这个操作叫做 **Escalate**（提权操作）。

在默认情况下，提权操作都是需要人工审核同意的。

### 三档权限模式

| 档位 | 模式 | 说明 |
|------|------|------|
| **第一档（默认）** | 需要审批 | 所有提权操作都需要人工审核 |
| **第二档（推荐）** | 自动审查 | Codex 自动调用一个小模型对提权操作进行安全性审查，低风险的自动放行，只有高风险的操作才会触发人工审查 |
| **第三档** | 完全访问权限 | Codex 完全无视沙箱的限制，可以在电脑上执行一切的操作 |

> ⭐ **推荐模式**：自动审查
> - 使得绝大部分操作不需要人工审批
> - 在获取了较高安全性的同时，还极大提升了使用的便利度

> ⚠️ **警告**：开启完全访问权限时，会出现很醒目的风险提示，提示要谨慎使用。

## 5. 上下文管理

### 上下文使用量

- 在权限控制的右边有一个圆圈，展示的是**当前上下文使用情况**
- 显示的是这个对话里的历史对话内容占用了多少模型上下文空间

### 上下文压缩

当上下文超过限制的时候，Codex 会自动对对话历史进行压缩，从而释放出更多的上下文空间。

**手动触发压缩**：
- 输入斜杠选择压缩选项
- 压缩完成以后，Codex 会把之前对话的一些不重要的内容排除掉
- 可以有效提高 AA 的专注力，并且降低 Token 消耗

### 通用经验

> 💡 **清空好于压缩**

因为过多的历史会话会干扰 AA 的注意力。

**最佳实践**：
- 当我们让 AA 执行完一个任务以后，最好是开一个新的对话，清空上下文
- 这样有助于 AA 把注意力全部集中到新的任务上面来，从而提高任务的执行效果

## 6. AI 画图功能

### 内置 AI 生图模型

- Codex 内置了 AA 画图功能
- 使用的是当今最强的 AA 生图模型：**GPT Image**（GBT 英魅之兔）

### 使用方法

1. 开启一个对话，选择项目
2. 输入提示词，让 Codex 调用 AI 绘图功能
3. Codex 生成图片后，可以自动替换到网页中

### 实际应用案例

**案例 1：生成店内环境轮播图**
```
让 Codex 调用 AI 绘图功能，
绘制三个店内环境的轮播图，
这三个图应该分别展示店内的不同区域
```

结果：Codex 生成了三张图片，基本都保持了店内装修风格的一致性，然后在网页中也替换成了三张图片的轮播图。

**案例 2：生成门店地图**

问题：原本用 SVG 生成的地图效果很差

解决方案：使用 Steer 功能引导 Codex 改用 AI 生图模型来画地图

## 7. Steer 引导功能

### 什么是 Steer？

- **英文名**：Steer
- **中文翻译**：引导
- **原意**：打方向盘

### 使用场景

当我们发现 AI 在执行过程中理解错了我们的意思，就不应该让它继续执行了，这时候应该及时接管方向盘，人工进行引导干预。

### 使用方法

**默认模式**：
- 新指令进入指令队列排队
- 需要 AI 把上一轮全部执行完，才能执行我们新输入的指令

**使用 Steer 功能**：
1. 点击"引导"按钮（英文版叫做 Steer）
2. 或者按快捷键 `Ctrl + Enter`
3. 中途接管方向盘，引导干预 AI 的执行

### 设置选项

在 Codex 的设置 → 常规设置里面，有一个"跟进行为"设置：

- **排队模式（默认推荐）**：在执行过程中，我们输入的指令是在后面排队
- **直接进入引导**：如果需要引导的话，直接点击引导按钮或者按快捷键

### 实际案例

**场景**：Codex 用 SVG 生成的地图效果很差

**操作步骤**：
1. 截个图发送给 Codex，说"你这图不行，应该调用 AI 绘图能力"
2. 点击"引导"按钮（Steer），中途接管方向盘
3. Codex 显示"已引导对话"，然后回复说"味不对，我立即改用 AI 生图，重新生成一张"
4. 使用 Steer 功能，在运行中途成功纠正了模型的运行方向

## 8. 计划模式（Plan Mode）

### 功能介绍

开启了计划以后，Codex 就不会立即上手干活，而是先为我们输出一份完整的工作计划，跟我们进行了确认以后再干活。

### 使用建议

> ⭐ **对于所有的复杂任务，建议都先开启计划模式**

确保你能跟 AI 对齐颗粒度，让 AI 能够精准理解我们的意图。

### 工作流程

1. 在 Codex 里面打开计划模式
2. 输入需求
3. Codex 会用问题卡片的形式跟用户进行沟通
   - 例如：希望使用哪种项目形态？（App Router + TS）
   - 样式迁移希望怎么处理？
   - 迁移完成要不要同时启动本地开发服务器验证？
4. Codex 生成一份完整的计划
5. 用户仔细阅读计划，确认没有问题
6. 点击"是，实施此计划"
7. 代码编写完成以后，Codex 可以启动它内置的浏览器进行自动化的测试

### 实际案例

**需求**：把项目改造成 Next.js 框架

**操作流程**：
1. 打开计划模式
2. 输入需求：把这个项目改造成 Next.js 框架
3. Codex 询问：
   - 希望使用哪种项目形态？→ 选择 App Router + TS
   - 样式迁移希望怎么处理？→ 选择改他要问的
   - 迁移完成要不要同时启动本地开发服务器验证？→ 选择构建加启动
4. Codex 生成完整计划
5. 点击"是，实施此计划"
6. 任务完成，开发服务器启动
7. 在右侧的多功能窗口，Codex 自动打开了浏览器
8. 点击"展开面板"按钮，查看项目的完整状态
9. 架构迁移非常成功，页面上所有的元素都完整的保留了

### 批注功能

在右侧多功能窗口的浏览器里：
1. 点击"批注"按钮
2. 选中一个元素
3. 在这里可以添加评论
4. 点击发送，让 AA 帮我们修改

## 9. 代码管理和 Git 操作

### Codex APP 并不是传统的 IDE

- 它并不提供完整的代码编辑功能
- 可以在右上角点击切换文件树，虽然可以查看代码，但是没法直接编辑
- 只能点击某行代码来写批注，并不能直接修改代码

### 使用第三方 IDE

我们可以借助第三方的 IDE 来修改代码：

1. 在进行代码管理之前，需要先把项目初始化成一个 Git 工程
   - 开启一个对话，输入提示词："把项目初始化成一个 Git 工程，注意排除掉不需要的文件"
   - Codex 先为我们创建了 `.gitignore` 文件，把一些不需要提交的内容排除出去
   - Codex 帮我们把项目初始化成了 Git 仓库

2. 初始化成 Git 仓库以后，右上角又多了很多按钮
   - 这里有一个 **VS Code 的按钮**，点击一下，就可以快捷的使用 VS Code 来查看和修改代码
   - 除了使用 VS Code，在这个下拉列表里面还可以使用其他的 IDE 来打开项目
   - 如果你的电脑上装过这些 IDE，就可以在这里关联出来
   - 我们也可以在设置 → 常规设置里面选择默认打开的 IDE

### GitHub 操作

我们可以要求 AA 把代码帮我们提交到 GitHub 上面：

1. 先在 GitHub 上面为它创建一个仓库
   - 点击"New"按钮
   - 填写仓库名字
   - 点击"创建"
   - 创建出来以后，把地址复制一下

2. 回到 Codex，把地址扔给它
   - Codex 成功帮我们把代码推送到了 GitHub 上面

3. 点击链接，看到代码已经备份到了 GitHub 的网站上面

> 💡 我们可以使用对话的方式，进行一切 Git 与 GitHub 操作。

## 10. Git Work Tree 并行开发

### 两个进阶技巧

1. 使用 Git 对开发过程进行回滚
2. Git Work Tree

### 技巧 1：使用 Git 进行回滚

#### 备份代码改动

当 AA 完成一个功能的开发以后，我们就使用 Git 把它备份保存一下：
1. 点击"提交"
2. 填写一个提交消息
3. 点击"继续"
4. 这样最新的代码改动就以 Git 的方式保存下来了

#### 回滚操作

如果觉得改动不好，想回滚：

**步骤 1：使用分叉（Fork）功能回退对话历史**
1. 找到上一次对话结尾的位置
2. 点击"分叉"按钮（英文是 Fork）
3. 选择"派生到本地"
4. Codex 的分叉功能就是在我点击的位置，把对话复制了一份
5. 这样复制出来的新对话，就已经剔除掉了我们刚才想删除的部分了

> ⚠️ 注意：分叉功能只能回退对话历史，它不能同步回退代码。

**步骤 2：使用 Git 回退代码**

分叉功能只能回退对话历史，它不能同步回退代码，所以这里我们要做的是把代码一同回退掉。

1. 在 VS Code 里面，点击"Source Control"按钮
2. 查看所有的 Git 提交
3. 找到需要回退到的提交
4. 点击右键，点击"Copy Commit Hash"
5. 这样就把这次提交的 ID 复制下来了
6. 回到 Codex，让 AA 把代码回退到这个状态
7. 代码回退成功

**结果**：我们使用了 Codex 对话分叉功能，加上 Git 操作，成功的把这一次不需要的改动，从代码层面和对话历史层面进行了完全的回滚。

---

### 技巧 2：Git Work Tree

#### 什么是 Work Tree？

- **Work Tree 名字听起来比较骇人**
- **本质上就是用 Git 创建一个新的分支，然后把这个新分支的代码完整的复制到一个新的文件夹里面**
- 这个新文件夹就是一个 Work Tree
- 主文件夹和分支文件夹可以并行工作
- 我们可以在两个文件夹里面各自修改代码，互相不干扰
- 我们可以基于主干创建多个分支，它们在底层通过 Git 关联在一起
- 分支文件夹的改动，随时都能轻松合并回主干

#### 使用方法

**创建 Work Tree**：

1. 找到我们的项目右键
2. 点击"创建永久工作树"
3. 给 Work Tree 起个名字
   - 例如：第一个工作树专注优化客户评价部分 → 加个后缀叫 Customer Rating
4. Codex 把整个项目复制到了一个单独的文件夹里面
5. 它跟主干已经不是同一个文件夹了
6. 我们在分支里面做的操作，不会影响到主干

可以创建多个 Work Tree：
- 第二个工作树主要用来负责优化下面的门店信息还有这里的地图
- 这样我们就拥有了两个工作树分支，它们都位于不同的文件夹下面
- 所以它们之间的并行工作，不会影响到主干

**在 Work Tree 里面开发**：

1. 打开第一个分支，新对话，输入需求
   - 例如：优化一下客户评价部分，多写几个评价，做一个动画轮播效果
2. 然后打开第二个分支，创建一个新的对话
   - 例如：优化一下门店信息部分，让门店信息与地图上下排列，不要左右排列，把地图展示全
3. 我们在两个分支上面进行并行开发
4. 因为它们位于两个不同的文件夹，所以互相之间不会产生干扰

**合并回主干**：

1. 两个分支在各自的文件夹里面都开发完毕了
2. 接下来我们可以把它合并回主干
3. 输入"合并回主干"
4. 两个分支都一样的操作，合并回主干
5. 两个分支都成功合并进了主干

**验证结果**：

在浏览器这边可以看到：
- 客户评价已经优化过了
- 门店信息也从左右排布变成了上下排布
- 地图展示的更全了

**清理 Work Tree**：

当我们的分支使用完毕：
1. 我们可以直接右键，点击"移除"
2. 把两个临时的分支移除掉
3. 然后回到主干继续工作

## 11. 云端运行环境

### 功能介绍

Codex 任务除了能在本地（也就是这台电脑上）运行，它也可以在云端运行，比如处理文件或者修改代码等等。

### 使用前提

启动云端运行环境的前提是：要先把代码全部同步到 GitHub 上面。

### 操作步骤

**步骤 1：推送代码到 GitHub**

1. 在右上角点击"推送"按钮
2. 点击"继续"
3. 这样我们所有的代码都同步到了 GitHub 上面
4. 确保了 GitHub 上面的远端代码跟本地代码是完全一致的

**步骤 2：关联 Codex Web**

1. 在运行模式这里选择"关联 Codex Web"
2. 也就是把这个项目先初始化成一个网页版可以用的项目
3. 右上角点击"前往 Cloud"
4. 接下来选择代码仓库
5. 这里输项目的名字
6. 然后直接在这里输入我们的需求就可以了

**步骤 3：在云端执行任务**

例如：让 Codex 给期望到店日期设置一个默认值，也就是明天早晨的 9 点半

执行过程：
1. 看到 Codex 先是初始化了一个云端运行环境
2. 然后把 GitHub 的代码下载进去了
3. 接着它调用 AA 功能，对整个项目进行分析
4. 然后完成了修改

**步骤 4：创建 Pull Request**

## 12. 记忆系统（agents.md）

### 问题背景

每当我们开启一个新对话，就进入了一个全新的上下文：
- AE 完全不记得之前发生了什么
- 进行过什么样的对话
- 甚至对于整个项目的记忆都是空白的

当项目变得复杂以后：
- 每次对话都要重新给 AE 交代一遍项目背景
- 或者让他自己读代码自己摸索
- 这是一种非常低效的工作方式

### 解决方案：agents.md 文件

#### 方法 1：项目级 agents.md

**通用方法**：在项目的根目录创建一个 `agents.md` 文件

- 这个文件在 Cloud Code、Open Code 等等其他的 AE Agent 工具里面也是通用的
- 这个文件就是 AE 每次对话的时候必读的一个指南
- 有了这个文件以后，后续我们跟 Codex 所有对话，都会带上这个文件的内容作为上下文
- 这个文件有助于帮助 AA 更快的理解项目

**使用方法**：

1. 用 VS Code 打开项目
2. 来到项目根目录右键，新建一个叫做 `agents.md` 的文件
3. 在文件里面补充上下文信息
   - 例如：我叫技术爬巴虾，擅长 Python 和 Java，对 CSS 一窍不通，遇到网页样式的问题需要用大白话给我解释

**验证**：

回到 Codex 新开一个对话，询问 Codex："我叫什么，擅长什么技术？"

看到 Codex 可以自动的读取 `agents.md` 文件，然后把里面的内容自动作为上下文带入对话。

**让 Codex 自动编写 agents.md**：

如果自己编写 `agents.md` 也有点麻烦，我们可以让 Codex 来帮我们编写：

1. 让 Codex 通读当前文件夹
2. 把它学到的关于项目的知识保存到 `agents.md` 文件里面
3. Codex 为我们完成了 `agents.md` 文件的编写
4. 把关于项目的重要知识都写入了这个文件
5. 后续当我们开启新对话的时候，Codex 就会自动的获取这些知识，可以帮助他更快速的上手项目

> 📌 **对于复杂项目来说，这个 `agents.md` 是必须要写的**

**作用范围**：

我们把 `agents.md` 放到项目的根目录，它**只对当前这个项目生效**。

#### 方法 2：全局 agents.md

Codex 提供了另外一种方式，可以编写**全局的 `agents.md`**，可以对这台电脑上所有的项目生效。

**设置方法**：

1. 回到 Codex 左下角设置
2. 在"个性化"这里找到"自定义指令"
3. 这里编辑的就是全局的 `agents.md`，也就是对所有项目都生效

**实际案例**：

最近看到一些朋友们说，Codex 有时候编写命令失误，把自己的整个地盘都删除了。

所以可以加上这么一个全局的提示词：
```
禁止它使用脚本批量删除文件或者目录，
只能使用 Remove-Item 一个文件进行删除。
如果必须批量删除文件，应该停止操作，让用户手动的批量删除。
```

这些内容保存进了 Codex 全局的 `agents.md` 文件里面。

**文件位置**：

我们可以来到 C 盘 → 用户 → 你的用户名 → `.codex` 这个文件夹，有一个 `agents.md` 文件，这里面就是我们刚才填写的，对所有项目生效的全局提示词。

#### 实验性特性：启用记忆

Codex 记忆功能还有一个实验性的特性，就是在下面有一个"启用记忆"：

- 开启以后，它可以从聊天里面生成新的记忆，并且把它带入新的聊天
- 因为这个功能目前还是一个实验性的功能，作者就不多做介绍了

## 14. MCP（模型上下文协议）

### 什么是 MCP？

- **全称**：Model Context Protocol（模型上下文协议）
- **简单来说**：MCP 就是 AA 大模型的标准化工具箱
- **功能**：大模型可以利用这些工具与外界互动，获取信息并且完成具体任务

### MCP 在 Codex 里面的入口

MCP 的入口隐藏的比较深：

### 实际案例：SuperBase MCP

#### SuperBase 介绍

SuperBase 是一个开源免费的 Postgres 云端数据库。

1. 来到个人主页

3. 然后我们在 SuperBase 这边找到 Connect，找到连接池
4. 把下面这个地址复制一下（注意要把这里替换成自己的密码）

**测试功能**：
